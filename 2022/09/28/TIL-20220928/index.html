오늘은~

git branching model들을 배웠습니다.
회사마다 관리하는 branching model이 다르고 이것도한 파악해 두는 것이 중요하다.
git flow 전략을 중점적으로 배웠습니다.
develop브랜치에서 개발한 것을 차곡차곡 모은 다음 정기적으로 release하는 방식으로 이런 싸이클의 제품군에 많이 적용합니다.
git-flow
웹 사이트를 이용해서 git-flow를 이해하고 사용하면 접근이 쉬워진다.

이렇게 하면 지금의 브랜치는 html-init이다.
html-init 브랜치 위에서 파일 만들고 편집하고 add, commit 한 뒤에

이러면 html-init 브랜치는 없어지고 develop 과 main 브랜치가 남게 됨.

이후에는 release과정을 거친다.

이러면 브랜치는 develop , main, release/v0.1 이 존재한다.

release 과정을 마치면 다시 develop 과 main 브랜치가 남게 됩니다.
마지막으로 remote 깃 환경에 push 할 차례 입니다.

push할 때는 기존에는 레포지터리의main브랜치를 클론해서 시작했기 때문에 main브랜치는 우리가 작업한 main과 remote의 main이 같은 것을 git이 알고 있는 것과 같은 원리로
develop브랜치가 완전히 같은 브랜치 임을 명시하기위해 upstream 인 -u를 사용합니다.
그런뒤 main으로 switch후 main도 마찬가지로 push 해줍니다.

git으로 작업환경 되돌리기.

  • working directory 에서 되돌리기
    • 이름바꾸기
      git mv 이전이름 새로운이름
    • undo
      git restore 파일
  • stage 에서 내리기
    git reset HEAD 파일
    git restore –staged 파일
  • commit 수정
    가장 최신의 수정은 괜찮지만 이미 지나간 과거의 커밋을 수정하는 것은 그렇게 권장하지 않는다.
  • commit 취소
    reset은 다 날릴 수 있어서 최대한 쓰지 않는 것이 좋다.
    revert를 쓰는 것을 추천. revert를 쓸 때는 돌아가려는 시점마다 커밋을 해야한다. 한번에 하려면 앞부분에 –no-commit을 붙인다.
    git revert –no-commit HEAD~3..(..까지 명령이다.)
